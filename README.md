# Proton - Web Game Engine

## Архитектура

Монорепо с пятью пакетами: `engine`, `stdlib`, `editor`, `server`, `shared`.

## Стек

- **Frontend** - React, Vite, Dockview, Zustand, Tailwind
- **Backend** - Fastify, chokidar, esbuild
- **Язык** - TypeScript везде

## ECS

Компоненты хранятся в `TypedArray` буферах (Float32Array, Int32Array и т.д.) по принципу SoA (Structure of Arrays). Entity это просто числовой ID.

Поля компонента описываются через декораторы:

```ts
class PositionComponent {
  @f32 x: number = 0;
  @f32 y: number = 0;
  @f32 z: number = 0;
}
```

Query организован через битовые маски. Каждый компонент получает уникальный бит при регистрации в мире. Маска entity хранится в массиве `Uint32Array` что позволяет поддерживать до N\*32 компонентов. Query поддерживает операторы `all`, `any`, `none`.

Entity ID использует generational подход - старшие биты это поколение, младшие это индекс. Это защищает от dangling references после recycling ID.

## Миры

Проект содержит миры. Мир это независимый контейнер сущностей и систем. Миры включаются и выключаются. Порядок и состояние миров настраивается в редакторе.

## Системы

```ts
class MySystem implements ISystem {
  onInit(): void {}
  onUpdate(delta: number): void {}
  onDestroy(): void {}
}

class MyRenderSystem implements IRenderSystem {
  onInit(): void {}
  onRender(encoder: GPUCommandEncoder): void {}
  onDestroy(): void {}
}
```

Системы помечаются декораторами:

- `@editor_only` - только в редакторе
- `@run_always` - в редакторе и в игре
- без декоратора - только в игре

## Ресурсы

Ссылочные типы (меши, текстуры, шейдеры, ссылки на entity) хранятся в `EmbeddedResourceManager` со счётчиком ссылок. В компонентах хранятся только числовые ID ресурсов. Ресурс выгружается когда счётчик ссылок достигает нуля.

## Файловая система

Проекты хранятся на диске по пути `имя_пользователя/имя_проекта/project_root`. Пользовательские скрипты написанные на TypeScript транспилируются через esbuild на сервере при сохранении файла и горячо перезагружаются в редакторе.
Каждый файл имеет свой теневой описательный файл .meta .
Принцип - все есть файл. Есть возможность создавать массивы, строки, классы и другие ссылочные данные в виде в файлов.

### Базы данных проекта

Каждый проект имеет две отдельные SQLite таблицы в базе данных проекта `project.db`:

**`settings`** - конфигурация и настройки проекта:

- Настройки миров и систем
- Конфигурация редактора
- Настройки сборки
- Прочая конфигурационная информация

**`permissions`** - система доступа и совместной разработки:

- `users` - глобальные пользователи системы
- `projects` - проекты
- `project_members` - пользователь в проекте, roleMask, personalAllow, personalDeny
- `roles` - роли проекта
- `file_permissions` - права на файл и директорию
- `file_locks` - заморозка объектов
- `audit_log` - лог действий

Пользовательские скрипты написанные на TypeScript транспилируются через esbuild на сервере при сохранении файла и горячо перезагружаются в редакторе.

## Групповая разработка

Система прав построена на битовых масках аналогично Linux.

**Права на файл и директорию:**

- `read` - просмотр в инспекторе
- `write` - изменение, удаление, перемещение
- `use` - применить в сцене / раскрыть директорию

Каждый файл и директория имеют три маски: `allowMask`, `denyMask`, `visibilityMask`.

**Приоритет прав:**

- Личный запрет > групповое разрешение
- Личное разрешение > групповой запрет

**Роли:**

- Суперпользователь - не принадлежит никому, активируется паролем проекта. Генерируется при создании проекта, хранится в `permissions.db`
- Модератор - бит в маске ролей, может передавать владение файлами
- Остальные роли - создаются владельцем проекта в специальной вкладке редактора

Один пользователь может участвовать в любом количестве проектов с разными ролями и разрешениями на каждом.

**Заморозка объектов:**

Пользователь может заморозить файл или объект сцены над которым работает. Замороженный объект доступен остальным только для чтения. Все изменения видны в реальном времени через WebSocket. Если пользователь отключился не сняв заморозку - снять её может суперпользователь или модератор.

## Редактор

Редактор построен на Dockview - оконная система с докингом, вкладками и resize. Основные панели: Scene, Hierarchy, Inspector, Console, FileSystem. Рендер сцены в редакторе использует WebGPU.
